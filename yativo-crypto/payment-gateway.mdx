The Crypto Payment Gateway lets you accept USDC and USDT payments on any supported chain. Use it as a **hosted checkout** (share a link, no frontend work) or integrate directly via the **API** into your own checkout flow. Funds settle into your Yativo-issued wallet — withdraw manually or enable [Auto-Forwarding](/yativo-crypto/auto-forwarding) to have them sent on automatically.

---

## How It Works

<Steps>
  <Step title="Create a payment link">
    Call `POST /crypto-gateway/payments` with the amount, title, and the list of assets you want to accept (e.g. `["USDC_SOL", "USDT_POL", "USDC_BASE"]`). You get back a payment ID, a hosted checkout URL, and an embed URL.
  </Step>
  <Step title="Customer chooses how to pay">
    On the hosted checkout page the customer picks from the payment methods you enabled, then gets the exact deposit address and a QR code to scan.
  </Step>
  <Step title="Payment is detected on-chain">
    Yativo monitors the deposit address. Once the transaction is confirmed on-chain the payment moves to `paid`.
  </Step>
  <Step title="Funds settle to your wallet">
    The settled amount is credited to your Yativo wallet. A webhook fires so your backend can fulfil the order.
  </Step>
</Steps>

## Create a Payment Link

```bash
curl -X POST 'https://crypto-api.yativo.com/api/v1/crypto-gateway/payments' \
  -H 'Authorization: Bearer YOUR_ACCESS_TOKEN' \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "Invoice #1234",
    "requested_amount": 49.99,
    "accepted_assets": ["USDC_SOL", "USDT_POL", "USDC_BASE"],
    "customer_email": "alice@example.com",
    "webhook_url": "https://yourapp.com/webhooks/payments",
    "external_reference": "order_abc123"
  }'
```

```json Response
{
  "success": true,
  "data": {
    "payment_id": "pay_a1b2c3d4e5f6",
    "title": "Invoice #1234",
    "requested_amount": 49.99,
    "status": "awaiting_payment",
    "checkout_url": "https://pay.yativo.com/pay/pay_a1b2c3d4e5f6",
    "embed_url": "https://pay.yativo.com/embed/pay_a1b2c3d4e5f6",
    "accepted_options": [
      { "asset_code": "USDC_SOL", "network": "solana", "deposit_address": "7xKXtg..." },
      { "asset_code": "USDT_POL", "network": "polygon", "deposit_address": "0xAbCd..." },
      { "asset_code": "USDC_BASE", "network": "base",   "deposit_address": "0x1234..." }
    ],
    "payment_window_expires_at": "2026-04-18T13:00:00Z",
    "monitoring_ends_at": "2026-04-18T13:20:00Z"
  }
}
```

Share the `checkout_url` with your customer. Each `accepted_options` entry has its own pre-provisioned deposit address — the customer picks one and pays to that address directly.

## Hosted Checkout Flow

The hosted page at `checkout_url`:

1. Shows the payment title and amount
2. Lists the payment methods you enabled (customer picks one)
3. Displays the deposit address and QR code for the chosen asset
4. Lets the customer enter their email to receive a receipt
5. Auto-refreshes status — shows a confirmation screen once detected on-chain

No integration code needed on your side beyond generating the link.

## Payment Statuses

| Status | Description |
|--------|-------------|
| `awaiting_payment` | Payment link created — waiting for the customer to send funds |
| `monitoring_late_window` | Primary window closed; payment still accepted for a short grace period |
| `paid` | Payment confirmed on-chain within the primary window |
| `paid_late` | Payment confirmed on-chain during the late monitoring window |
| `expired` | Monitoring window closed with no payment detected |
| `cancelled` | Cancelled by the merchant |

## Settlement

Once a payment is detected:

1. The received amount (minus platform fees) is credited to your **Yativo-issued wallet** for that asset.
2. If you have [Auto-Forwarding](/yativo-crypto/auto-forwarding) enabled on that wallet, funds are automatically forwarded to your configured external address.
3. Otherwise, the balance sits in your Yativo wallet and you can withdraw at any time.

The `merchant_settlement` object on the payment record tracks credit status, credited amount, and whether auto-forwarding was triggered.

## Compliance Gate

Before your first payment link, you must submit a Gateway Application. This is a one-time step.

```bash
curl -X POST 'https://crypto-api.yativo.com/api/v1/crypto-gateway/apply' \
  -H 'Authorization: Bearer YOUR_ACCESS_TOKEN' \
  -H 'Content-Type: application/json' \
  -d '{
    "logo_url": "https://yoursite.com/logo.png",
    "support_email": "support@yoursite.com",
    "social_media_url": "https://twitter.com/yourcompany",
    "business_type": "ecommerce",
    "expected_volume": "1k_10k",
    "website_url": "https://yoursite.com",
    "webhook_url": "https://yoursite.com/webhooks/yativo"
  }'
```

| Field | Required | Description |
|-------|----------|-------------|
| `logo_url` | Yes | HTTPS URL to your brand logo — shown on all checkout pages |
| `support_email` | Yes | Customer-facing support email — included in expired checkout links |
| `social_media_url` | Yes | Company social media URL for identity verification |
| `business_type` | Yes | One of: `ecommerce` `saas` `marketplace` `freelance` `ngo` `gaming` `travel` `finance` `other` |
| `expected_volume` | Yes | One of: `under_1k` `1k_10k` `10k_50k` `50k_250k` `over_250k` |
| `website_url` | No | Your business website |
| `webhook_url` | No | Default webhook for all payment events — can be updated by reapplying |

Once submitted, your application enters `pending` status. Attempting to create payment links before approval returns a `403` with `error_code: GATEWAY_APPLICATION_REQUIRED`.

```bash
# Check your application status
curl 'https://crypto-api.yativo.com/api/v1/crypto-gateway/application' \
  -H 'Authorization: Bearer YOUR_ACCESS_TOKEN'
```

## Hosted Checkout Flow

The checkout page at `checkout_url`:

1. If multiple assets are accepted, the customer picks one
2. The customer enters their email (skipped if `customer_email` was passed at creation)
3. The deposit address and QR code are revealed
4. The customer sends the **exact** asset, on the **exact** network, for the **exact** amount shown in checkout
5. The page polls for status and auto-transitions to a success screen once detected on-chain

If the payment window expires before funds arrive, the customer sees an expired state with a "I already sent crypto — report it" button that opens the [underpayment report form](#underpayment-reconciliation). The hosted checkout also warns customers not to send the wrong token or wrong network, for example `USDT` instead of `USDC`, because those cases require manual recovery handling.

---

## Settlement Rules

| Scenario | Outcome |
|----------|---------|
| Exact or over-payment received on-chain within the primary window | Settled as `paid` — merchant credited full on-chain amount |
| Payment arrives in the late monitoring window | Settled as `paid_late` — same credit rules |
| Underpayment (received < requested) | Settlement rejected — merchant must use the reconciliation flow |
| Payment arrives after monitoring window | Rejected unless `force_settle: true` is passed on the internal settle endpoint |

Once settled, the net amount (on-chain amount minus platform fees) is credited to the merchant's Yativo-issued wallet for that asset. If [Auto-Forwarding](/yativo-crypto/auto-forwarding) is enabled, it triggers automatically.

## Underpayment Reconciliation

If a customer paid but the amount was short and the window expired, both sides can resolve it without a support ticket.

### Customer side (checkout page)

On the expired checkout, the customer clicks **"I already sent crypto — report it"** and submits:
- Their sending wallet address (required)
- How much they sent (optional)
- Their email (optional, if not already captured)
- A note to the merchant (optional), for example explaining that they sent the wrong token or wrong network

```bash
# This endpoint is public — no auth required
curl -X POST 'https://crypto-api.yativo.com/api/v1/crypto-gateway/pay/pay_a1b2c3d4e5f6/report-underpayment' \
  -H 'Content-Type: application/json' \
  -d '{
    "customer_sending_wallet": "7xKXtgMK...",
    "reported_amount": 45.00,
    "customer_email": "alice@example.com",
    "customer_note": "I accidentally sent USDT instead of USDC. Please escalate for recovery review."
  }'
```

You receive a `gateway.underpayment.reported` webhook immediately.

### Merchant side (dashboard or API)

```bash
# List reports
GET /crypto-gateway/reconciliation?status=pending

# Step 1 — verify the on-chain balance at the gateway wallet
POST /crypto-gateway/reconciliation/:reportId/verify

# Optional — escalate wrong-token or wrong-network cases to platform recovery
POST /crypto-gateway/reconciliation/:reportId/escalate-recovery

# Step 2a — reverse: queue a refund using admin-configured fee
POST /crypto-gateway/reconciliation/:reportId/reverse

# Step 2b — reconcile: create a new payment link for the shortfall
POST /crypto-gateway/reconciliation/:reportId/reconcile

# Step 2c — settle as paid: revise original requested amount to verified amount and settle
POST /crypto-gateway/reconciliation/:reportId/settle-as-paid
```

**Verify** queries the live on-chain balance at the original gateway wallet and stores `on_chain_verified_amount`.

**Escalate Recovery** is the merchant action for wrong-token or wrong-network cases. It creates an admin dashboard alert, emails all admins, sends a Telegram alert, and pushes ops into the existing cross-token recovery workflow.

**Reverse** queues an ops-processed on-chain refund of the verified amount minus the merchant's admin-configured underpayment reversal fee. The fee is configured in admin gateway settings per merchant.

**Reconcile** creates a new `GatewayPaymentIntent` for exactly `requested_amount − on_chain_verified_amount`. When the customer pays this reconciliatory link, the platform **credits both amounts to the merchant together** — the original verified balance plus the top-up — completing the full order. The original payment intent appears in the new intent's audit trail.

**Settle As Paid** lets the merchant complete the original payment intent immediately using the verified on-chain amount. The original requested amount is revised to the verified amount, settlement runs on the original intent, and an immutable audit trail entry is added to the report.

If the payer note indicates a wrong token or wrong network, the merchant should escalate it to Yativo ops instead of settling or refunding immediately. Those cases are handled through the existing cross-chain recovery process.

## Reconciliation Report Statuses

| Status | Description |
|--------|-------------|
| `pending` | Submitted by customer — awaiting merchant verification |
| `verified` | On-chain balance confirmed — merchant chooses next action |
| `reversing` | Refund queued — ops processing the on-chain transfer |
| `reversed` | Refund sent on-chain |
| `reconciled` | Reconciliatory payment intent created (or settled) |
| `settled_adjusted` | Original intent settled after revising requested amount to paid amount |
| `dismissed` | Dismissed by merchant or admin |
