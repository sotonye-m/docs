---
title: "Secure Card Display (PSE iframe)"
description: "Display sensitive card details and PIN using tokenized secure iframes — card data and PIN each require separate tokens"
---

<Note>
  Card details (PAN, CVV, expiry) and the card PIN each use **separate ephemeral tokens**. You must make two distinct API calls — one per display type — and never reuse tokens between them.
</Note>

## How it works

Sensitive card data is never returned in plain API responses. Instead, Yativo uses a **PSE (Partner Secure Element) iframe** pattern:

1. Your backend calls the ephemeral-token endpoint, specifying which view to render (`cardData` or `cardPin`)
2. Yativo returns a short-lived token scoped to that view only
3. Your frontend loads the PSE SDK and mounts a sandboxed iframe using that token
4. The iframe renders the sensitive data directly from Yativo's secure environment — your application never touches the raw values

<Warning>
  Ephemeral tokens expire in **60 seconds**. Never log, cache, or store them. Always request a fresh token immediately before rendering.
</Warning>

---

## Token types

| View | Element Type | What it displays |
|---|---|---|
| Card details | `cardData` | Full card number (PAN), CVV, expiry date |
| View PIN | `cardPin` | Current card PIN |
| Set PIN | `setCardPin` | PIN change form |

Card data and PIN tokens are issued by the same endpoint but are **distinct tokens** — they have different scopes and cannot be swapped. Always call the endpoint fresh for each view.

## Step 1 — Get a Card Data Token

Call this from your **backend** (never from the browser). The token is single-use and scoped to `cardData`.

```
POST /yativo-card/{yativoCardId}/cards/{cardId}/ephemeral-token
```

<ParamField path="yativoCardId" type="string" required>
  The Yativo Card account ID (`yativo_card_id`) from onboarding.
</ParamField>

<ParamField path="cardId" type="string" required>
  The card ID from card creation.
</ParamField>

<ParamField body="element_type" type="string" required>
  Use `"cardData"` to request a card details token. Use `"cardPin"` for PIN display.
</ParamField>

<RequestExample>
```bash Card Data Token
curl -X POST 'https://crypto-api.yativo.com/api/yativo-card/yc_01HX9KZMB3F7VNQP8R2WDGT4E5/cards/card_01HX9KZMB3F7VNQP8R2WDGT4E5/ephemeral-token' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{ "element_type": "cardData" }'
```

```bash PIN Token
curl -X POST 'https://crypto-api.yativo.com/api/yativo-card/yc_01HX9KZMB3F7VNQP8R2WDGT4E5/cards/card_01HX9KZMB3F7VNQP8R2WDGT4E5/ephemeral-token' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{ "element_type": "cardPin" }'
```
</RequestExample>

<Accordion title="Token Response Type Definitions">
```typescript
interface EphemeralTokenResponse {
  ephemeral_token: string;      // Short-lived PSE session token — scoped to one element type
  app_id: string;               // PSE application identifier
  card_token: string;           // Card-specific token passed to gp.init() — different per element type
  pse_auth_token: string;       // PSE auth token for SDK initialization
  last_four: string;            // Last 4 digits of the card (safe to display)
  pse_config: {
    sdk_url: string;            // Pinned PSE SDK CDN URL
    allowed_element_types: string[];  // What this token can render: ["cardData"] or ["cardPin"]
  };
  expires_at: string;           // ISO 8601 timestamp — token is invalid after this
}
```
</Accordion>

## Step 2 — Load the PSE SDK and Mount the iframe

The PSE SDK is loaded from a pinned CDN URL returned in the token response (`pse_config.sdk_url`). Always use this URL — do not hardcode a different version.

The SDK mounts a sandboxed iframe into a container element you provide. Your application never sees the raw card number or PIN.

<Tabs>
  <Tab title="Vanilla JS">
```html
<!-- 1. Container element for the secure iframe -->
<div id="card-details-container" style="min-height: 280px;"></div>

<script type="module">
  // 2. Fetch a fresh token from YOUR backend (not from the browser directly)
  async function getCardDataToken() {
    const res = await fetch('/api/card-token', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ element_type: 'cardData' })
    });
    if (!res.ok) throw new Error('Failed to fetch token');
    return res.json(); // { ephemeral_token, app_id, card_token, pse_auth_token, ... }
  }

  // 3. Load PSE SDK and mount iframe
  async function mountCardDetails() {
    const token = await getCardDataToken();

    // Validate token shape before trusting it
    if (!token.ephemeral_token || !token.app_id || !token.card_token) {
      throw new Error('Invalid token response');
    }

    // Verify token is for the correct element type
    if (!token.pse_config?.allowed_element_types?.includes('cardData')) {
      throw new Error('Token is not valid for cardData display');
    }

    // Load SDK from the URL returned in the token (do not hardcode)
    const { default: GPSDK, ElementType } = await import(token.pse_config.sdk_url);

    const gp = new GPSDK({
      appId: token.app_id,
      ephemeralToken: token.ephemeral_token,
      gnosisPayApiAuthToken: token.pse_auth_token,
      onActionSuccess: (action) => console.log('PSE action succeeded:', action),
      onInvalidToken: () => mountCardDetails(), // auto-refresh on token expiry
      onError: (msg) => console.error('PSE error:', msg),
    });

    // Mount the iframe — gp.init() injects a sandboxed iframe into your container
    gp.init(ElementType.CardData, '#card-details-container', {
      cardToken: token.card_token,
    });
  }

  mountCardDetails().catch(console.error);
</script>
```

import { useEffect, useRef } from 'react';
import { YativoCardEmbed } from '@yativo/crypto-sdk/card-embed';

interface SecureCardDisplayProps {
  yativoCardId: string;
  cardId: string;
  lastFour: string;
  cardHolderName: string;
  view: 'data' | 'pin' | 'setpin';
}

export function SecureCardDisplay({
  yativoCardId,
  cardId,
  lastFour,
  cardHolderName,
  view,
}: SecureCardDisplayProps) {
  const containerRef = useRef<HTMLDivElement>(null);
  const widgetRef = useRef<ReturnType<typeof YativoCardEmbed.mount> | null>(null);

  useEffect(() => {
    if (!containerRef.current) return;

    widgetRef.current = YativoCardEmbed.mount(containerRef.current, {
      onTokenRequest: async () => {
        // This must call YOUR backend — never expose API credentials in the browser
        const res = await fetch('/api/card-token', {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({ yativoCardId, cardId, element_type: view === 'data' ? 'cardData' : 'cardPin' }),
        });
        return res.json();
      },
      lastFour,
      cardHolderName,
      enabledViews: [view],
    });

    return () => {
      widgetRef.current?.destroy();
    };
  }, [yativoCardId, cardId, view]);

  return <div ref={containerRef} style={{ minHeight: 380 }} />;
}

## Step 3 — Backend token proxy endpoint

Your frontend must **never** call the Yativo API directly with your API credentials. Always proxy token requests through your backend:

```typescript
// Example: Express.js proxy endpoint
import express from 'express';

const app = express();
app.use(express.json());

app.post('/api/card-token', async (req, res) => {
  // Authenticate your own user first
  const user = await authenticateRequest(req);
  if (!user) return res.status(401).json({ error: 'Unauthorized' });

  const { element_type } = req.body;

  // Validate element_type — only accept known values
  const VALID_TYPES = ['cardData', 'cardPin', 'setCardPin'];
  if (!VALID_TYPES.includes(element_type)) {
    return res.status(400).json({ error: 'Invalid element_type' });
  }

  // Look up this user's card IDs from your own database
  const { yativoCardId, cardId } = await getUserCardIds(user.id);

  // Call Yativo API from your backend — credentials never leave the server
  const response = await fetch(
    `https://crypto-api.yativo.com/api/yativo-card/${yativoCardId}/cards/${cardId}/ephemeral-token`,
    {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${process.env.YATIVO_API_KEY}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({ element_type }),
    }
  );

  if (!response.ok) {
    return res.status(502).json({ error: 'Failed to get token' });
  }

  const tokenData = await response.json();

  // Return the token data to the frontend
  // The token expires in 60s — do not add extra caching
  res.json(tokenData.data);
});
```

## Security requirements

<AccordionGroup>
  <Accordion title="Token isolation — never mix card data and PIN tokens">
    Each call to the ephemeral-token endpoint returns a token scoped to **one** element type, reflected in `pse_config.allowed_element_types`. Always check this before initializing the SDK:

    ```javascript
    // Good — verify before use
    if (!token.pse_config?.allowed_element_types?.includes(requestedType)) {
      throw new Error(`Token not valid for ${requestedType}`);
    }
    ```

    Never pass a `cardData` token when mounting `CardPin`, or vice versa — the SDK will reject it.
  </Accordion>
  <Accordion title="No inline scripts — Content Security Policy">
    The PSE SDK loads via dynamic `import()`. Ensure your CSP allows the CDN origin and does not use `unsafe-inline` for scripts:

    ```http
    Content-Security-Policy:
      script-src 'self' https://unpkg.com;
      frame-src 'self' https://secure.gnosispay.com;
      object-src 'none';
    ```
  </Accordion>
  <Accordion title="Container sanitization">
    The PSE SDK writes only to the container element you pass. Keep that container isolated — do not inject other untrusted content into the same element:

    ```javascript
    // Safe: dedicated container, only PSE SDK writes here
    const container = document.getElementById('card-details-container');
    container.innerHTML = ''; // clear before mounting
    gp.init(ElementType.CardData, '#card-details-container', { cardToken });
    ```
  </Accordion>
  <Accordion title="No token logging or storage">
    ```javascript
    // NEVER do this
    console.log(token); // logs ephemeral_token
    localStorage.setItem('cardToken', token.card_token); // stored in browser

    // ALWAYS do this instead — use immediately, discard after use
    const token = await getCardDataToken();
    gp.init(elementType, '#container', { cardToken: token.card_token });
    // token goes out of scope here
    ```
  </Accordion>
  <Accordion title="Auto-refresh on token expiry">
    The SDK fires `onInvalidToken` when a token expires mid-session. Always handle this by fetching a fresh token:

    ```javascript
    const gp = new GPSDK({
      // ...
      onInvalidToken: () => {
        // Fetch a fresh token and re-initialize
        mountCardDetails();
      },
    });
    ```
  </Accordion>
</AccordionGroup>

## iframe behavior reference

| Event | Default behavior | Override via |
|---|---|---|
| Token expires | Calls `onInvalidToken` → auto-refresh | `onInvalidToken` callback |
| Display success | Calls `onActionSuccess` | `onActionSuccess` callback |
| SDK error | Calls `onError`, shows retry button | `onError` callback |
| Tab switch (`data` ↔ `pin`) | Fetches a new token for the new view | `onTokenRequest` |

<Note>
  Switching between **Card Data** and **PIN** tabs always triggers a fresh `onTokenRequest` call — each tab uses a different token scoped to its element type. Your `onTokenRequest` implementation should handle both element types (or pass the current `view` to your backend).
</Note>
